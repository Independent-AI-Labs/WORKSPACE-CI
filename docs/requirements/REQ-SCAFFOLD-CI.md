## 1. Purpose & Scope

### 1.1 Purpose

Eliminate the hand-crafted `.pre-commit-config.yaml` and `Makefile`
authoring step for new consumer projects by providing a single
profile-driven generator that emits a complete, consistent CI integration
package with paths computed correctly relative to the consumer-to-CI
filesystem offset.

### 1.2 Scope

**This document OWNS the requirements for:**

- The `ci-profile.yaml` per-consumer declaration file (presence, schema,
  validation rules, lifecycle).
- The `scripts/scaffold-ci` generator script (CLI, phases, output
  semantics, failure modes).
- The `CI/templates/ci-profile.template.yaml` reference template
  (contents, regeneration workflow).
- The generated `.pre-commit-config.yaml` (format, entry-rendering per
  hook `kind`, override application rules).
- The generated `Makefile` (contract compliance, vacuous-pass semantics,
  `CI_DIR` resolution).
- The copy of per-project-overridable configs from `CI/config/` into
  `<consumer>/config/` (which files, substitution rules, overwrite
  policy).
- The rendered `quality_exceptions.yaml` (template, overwrite policy).
- The `scaffold-ci` Makefile target in CI's own `Makefile`.
- The `scaffold-ci` entry in `scripts/manifest.yaml`.
- Unit and integration test coverage for the above.

**This document DOES NOT:**

- Own `generate-hooks` semantics (see HOOKS.md).
- Own the `makefile_contract.mk` target list (that file owns it; the
  generator complies with it).
- Own the `required_hooks.yaml` schema (that file's own schema file
  owns it; the generator reads it as input).
- Require changes to `banned_words.yaml`, `silent_swallow_patterns.yaml`,
  `sensitive_files.yaml`, `blocked_commit_patterns.yaml`,
  `boot_layout.yaml`, or any of the `*_schema.yaml` files: those stay
  workspace-policy, owned by CI.
- Mandate `moon.yml` generation: each project's moon task graph is
  per-project; consumers hand-author `moon.yml`.
- Touch the `project_enforcement.yaml` workspace-tier registry: it is
  per-machine, autocreated per SPEC-BOOT-LAYOUT §10.

### 1.3 Glossary

| Term | Definition |
|------|------------|
| Consumer project | Any repository under `projects/` (or any sibling of `CI`) that wants WORKSPACE-CI's git hooks, contract Makefile, and per-project config files scaffolded. |
| CI profile | A YAML file at `<consumer>/ci-profile.yaml` declaring the project's name, languages, enforcement tier, and the ordered list of hook IDs per stage. The single source of truth for "what CI gates this project runs." |
| Profile template | `CI/templates/ci-profile.template.yaml` -- the reference profile containing every hook from `required_hooks.yaml`. A LIVING document, regenerated by `--emit-template`. Consumers copy it as their `ci-profile.yaml` and trim. |
| Master hook registry | `CI/config/required_hooks.yaml` -- the workspace-wide canonical list of hooks with metadata. READ-ONLY by the generator. |
| Relative CI offset (`REL_CI`) | The single computed path constant: the relative path from the consumer's directory to `CI_ROOT` (e.g. `../CI`, `../../CI`). Computed by `realpath --relative-to`. Substituted into every generated file. |
| Mandatory hook | A hook in `required_hooks.yaml` with `mandatory: true`. The generator auto-inserts missing mandatory hooks (per the profile's declared tier) at the end of their stage list. |
| Safety hook | A hook with `safety: true`: installed even at `poc` tier. |
| Applicable hook | A hook whose `applicable_to` list intersects the profile's `languages` list. The generator validates non-applicable hooks (warning, not failure). |
| Auto-insertion | Generator behaviour: append a missing mandatory hook to the end of its stage list with a printed notice. Never reorders user-listed hooks. |
| Override | Per-hook-entry substitution declared in the profile's `overrides:` block. Limited to `entry`, `files`, `exclude`, `always_run`, `pass_filenames`. |
| Maintenance mode | `--emit-template` flag: regenerates `templates/ci-profile.template.yaml` from `required_hooks.yaml` and exits. Bypasses all consumer-side generation. |
| Vacuous pass target | A generated Makefile target whose recipe body is `@:` (bash no-op, exit 0). Lets `make check` pass out-of-the-box on a brand-new project without forcing the consumer to wire language-specific tooling before their first commit. |

### 1.4 Lifecycle Position

```
  ci-profile.yaml
        │
        ▼
   scaffold-ci ──► .pre-commit-config.yaml
        │             Makefile
        │             config/*.yaml
        │             quality_exceptions.yaml
        ▼
  (consumer commits the generated artifacts)
        │
        ▼
  make install-hooks
        │
        ▼
  generate-hooks ──► .git/hooks/{pre-commit,commit-msg,pre-push}
```

`scaffold-ci` produces the input for `generate-hooks`. The two scripts
are independent and rerunnable. `scaffold-ci` does NOT call
`generate-hooks`; the consumer runs `make install-hooks` separately
(which does call it).

## 2. Functional Requirements

### 2.1 CI Profile File

#### FR-SC-1: Profile Existence and Location

| ID | Requirement |
|----|-------------|
| FR-SC-1.1 | The consumer's working profile MUST live at `<consumer>/ci-profile.yaml`. |
| FR-SC-1.2 | The generator MUST accept a `--profile PATH` override pointing at a profile location other than `<consumer>/ci-profile.yaml`. |
| FR-SC-1.3 | If neither the default profile path nor a `--profile`-supplied path exists on disk, the generator MUST fail fast with a non-zero exit code and a printed diagnostic. The generator MUST NOT create a profile from scratch (consumers copy the template). |
| FR-SC-1.4 | `ci-profile.yaml` MUST be committed to the consumer's repo (per SPEC §16 Q6 -- checked-in is the per-repo source of truth). The generator does not enforce this; it is a social contract via PR review. |

#### FR-SC-2: Profile Schema

| ID | Requirement |
|----|-------------|
| FR-SC-2.1 | The profile MUST declare `version: 1` as the top-level version field. Other versions are rejected. |
| FR-SC-2.2 | The profile MUST declare `project: <string>` -- the project name rendered into `quality_exceptions.yaml`. The value MUST match `^[A-Za-z0-9][A-Za-z0-9_.-]*$` (alphanumeric start plus limited punctuation; no whitespace, no shell metacharacters). |
| FR-SC-2.3 | The profile MUST declare `languages:` as a non-empty YAML list. Each entry MUST be one of `any`, `python`, `rust`, `node`. The sentinel `any` MUST NOT coexist with other language entries. |
| FR-SC-2.4 | The profile MUST declare `tier:` as one of `strict`, `poc`, `vendored`. |
| FR-SC-2.5 | When `tier != vendored`, the profile MUST declare `hooks:` as a map keyed by stage name. Stage keys MUST be a subset of `pre-commit`, `commit-msg`, `pre-push`. Any other key is a validation failure. |
| FR-SC-2.6 | When `tier == vendored`, the profile MUST NOT declare a `hooks:` block. Its presence is a validation failure (signals the profile author may have intended `strict`). |
| FR-SC-2.7 | The profile MAY declare an `overrides:` map. Keys MUST be hook IDs already listed in the profile's `hooks:` block. Override field names MUST be a subset of `entry`, `files`, `exclude`, `always_run`, `pass_filenames`. Other fields are rejected. |
| FR-SC-2.8 | The profile's complete schema lives at `CI/config/ci-profile.schema.yaml` (JSON Schema Draft-4, matching the existing schemas in `config/*.schema.yaml`). |

### 2.2 Generator

#### FR-SC-3: CLI Surface

| ID | Requirement |
|----|-------------|
| FR-SC-3.1 | The generator MUST accept `--consumer PATH` (required for normal operation). PATH MUST be an existing directory; the trailing slash is stripped before use. |
| FR-SC-3.2 | The generator MUST accept `--profile PATH` (optional; default is `<consumer>/ci-profile.yaml`). |
| FR-SC-3.3 | The generator MUST accept `--dry-run` (print intended file contents to stdout; do NOT write any file). |
| FR-SC-3.4 | The generator MUST accept `--force` (overwrite existing `.pre-commit-config.yaml` and `Makefile` unconditionally; does NOT overwrite `quality_exceptions.yaml` per FR-SC-9.2; does NOT overwrite already-customised `config/` files per FR-SC-8.3 rule 3). |
| FR-SC-3.5 | The generator MUST accept `--emit-template` (maintenance mode: regenerate `CI/templates/ci-profile.template.yaml` from `CI/config/required_hooks.yaml` and exit; ignore all other flags). |
| FR-SC-3.6 | The generator MUST accept `-h` / `--help` and print usage synopsis. |
| FR-SC-3.7 | Unknown flags MUST cause the generator to fail fast with a non-zero exit code and a diagnostic. |
| FR-SC-3.8 | The generator MUST run without root; no `sudo` in any code path. |

#### FR-SC-4: Validation Phases

| ID | Requirement |
|----|-------------|
| FR-SC-4.1 | The generator MUST perform validation BEFORE any file is written. Failure in any phase exits non-zero without touching disk (no partial writes). |
| FR-SC-4.2 | Phase A (schema parse): the profile YAML MUST parse without error and contain `version: 1`. Failure: exit 1 with parser diagnostic. |
| FR-SC-4.3 | Phase B (required-field presence): the generator MUST collect ALL missing required fields and fail with a complete list (not fail-on-first). |
| FR-SC-4.4 | Phase C (field-value validation): the generator MUST validate field values per the schema (FR-SC-2). All errors accumulated; failure surfaces a list. |
| FR-SC-4.5 | Phase D (hook ID resolution): every hook ID listed in `hooks.<stage>` MUST exist in `CI/config/required_hooks.yaml`. Unknown IDs fail with the exact diagnostic `FAILED: hook '$ID' listed in stage '$STAGE' of ci-profile.yaml is not registered in config/required_hooks.yaml.` |
| FR-SC-4.6 | Phase D (applicability warning): a hook whose `applicable_to` does not intersect the profile's `languages` MUST emit a `ci_warn` but MUST NOT fail. The user may legitimately want an inapplicable hook (e.g., a Rust project running the Python `check-init-files` for cross-language discipline). |
| FR-SC-4.7 | Phase E (mandatory completeness): for the declared `tier`, every mandatory hook applicable to the declared `languages` MUST be in the profile's `hooks` block. Missing mandatory hooks are auto-inserted at the END of their natural stage (the stage listed in `required_hooks.yaml`) with a printed `ci_warn` listing each auto-insertion. The generator NEVER reorders user-listed hooks; it only appends. |
| FR-SC-4.8 | When `tier == vendored`: the generator MUST skip all generation and exit 0 with the notice `INFO: tier=vendored: no CI integration generated.` |

#### FR-SC-5: Path Computation

| ID | Requirement |
|----|-------------|
| FR-SC-5.1 | The generator MUST compute `REL_CI = realpath --relative-to="$CONSUMER_DIR" "$CI_ROOT"` exactly once per run. |
| FR-SC-5.2 | `$CI_ROOT` MUST be resolved from the script's own location (not from `pwd` or user argument): `_CI_ROOT="$(cd "$(dirname "$0")/.." && pwd)"`. |
| FR-SC-5.3 | If `realpath(1)` is missing from PATH, the generator MUST fail fast with a non-zero exit code. No silent fallback to `readlink -f` heuristics or manual path-construction string manipulation (per AGENTS.md Rule 13). |
| FR-SC-5.4 | The generator MUST operate on a non-git consumer directory (realpath is filesystem-pure; no `git rev-parse --show-toplevel` dependency in the path-computation phase). |

### 2.3 Generated `.pre-commit-config.yaml`

#### FR-SC-6: Format and Entry Rendering

| ID | Requirement |
|----|-------------|
| FR-SC-6.1 | The generated file MUST follow the EXACT structure of WORKSPACE-GUARD's existing `.pre-commit-config.yaml`: a single `repos:` block containing one `- repo: local` entry with a flat `hooks:` YAML list of `- id:` blocks. |
| FR-SC-6.2 | Each hook's `entry:` MUST be rendered according to its `kind` from `required_hooks.yaml` per the table in SPEC §6.2. Specifically: `shell` → `bash -c 'source <REL_CI>/lib/checks.sh && <entry>'`; `shell_inline` → verbatim `<entry>`; `shell_with_arg` → `bash -c 'source <REL_CI>/lib/checks.sh && <entry> "$1"' --`; `python_module` → `<REL_CI>/.venv/bin/python -m <entry>`; `python_module_files` → `<REL_CI>/.venv/bin/python -m <entry> "$@"`; `makefile_target` → `make <entry>` (no `<REL_CI>` substitution). |
| FR-SC-6.3 | The relative path `<REL_CI>` MUST be substituted into shell-source `entry:` templates only. `makefile_target` entries run from the consumer's repo root and rely on the consumer's own `Makefile` to resolve `CI_DIR` internally. |
| FR-SC-6.4 | The fields `pass_filenames`, `always_run`, `files`, `exclude` MUST be copied verbatim from `required_hooks.yaml` (after override application). |
| FR-SC-6.5 | Overrides from the profile's `overrides:` block MUST be applied BEFORE the path-substitution rule. If the user supplies an `entry:` override, the generator uses it verbatim (no `<REL_CI>` substitution). The user is responsible for embedding the correct relative path in the override. |
| FR-SC-6.6 | The file MUST be preceded by a comment block documenting: the generator source path, the timestamp, the tier and languages, and the re-generation command. |
| FR-SC-6.7 | If `required_hooks.yaml` is missing, the generator MUST fail fast with `FAILED: config/required_hooks.yaml not found at <CI_ROOT>/config/.`. No generation occurs. |
| FR-SC-6.8 | If the profile's `kind` field references an unknown value (not one of the six kinds above), the generator MUST fail with the offending hook ID and the unrecognised kind. Adding new kinds requires a code change to `scaffold-ci`. |

### 2.4 Generated `Makefile`

#### FR-SC-7: Contract Compliance and Stub Semantics

| ID | Requirement |
|----|-------------|
| FR-SC-7.1 | The generated `Makefile` MUST define all 10 mandatory targets from `lib/makefile_contract.mk`: `init`, `install`, `install-ci`, `install-hooks`, `sync`, `check`, `lint`, `type-check`, `test`, `clean`, `preflight`. |
| FR-SC-7.2 | The generated `Makefile` MUST `-include $(CI_DIR)/lib/makefile_contract.mk` where `CI_DIR := $(abspath $(REPO_ROOT)/<REL_CI>)`. This provides `make contract-check` and validates the 10 targets' presence via `make -n`. |
| FR-SC-7.3 | The language-specific targets (`lint`, `type-check`, `test`, `check-push`) MUST be vacuous pass targets with `@:` recipe bodies and `TODO` echo diagnostics. They MUST exit 0 so `make check` passes out-of-the-box on a brand-new project. |
| FR-SC-7.4 | The `install-hooks` target MUST call `bash $(CI_DIR)/scripts/generate-hooks`. |
| FR-SC-7.5 | The `clean-precommit` target MUST call `bash $(CI_DIR)/scripts/cleanup-precommit`. |
| FR-SC-7.6 | The generator MUST NOT generate `moon.yml` (per SPEC §1.3 -- out of scope). |
| FR-SC-7.7 | The generated `Makefile` MUST include a `help` target that greps `## ` annotations and prints them, mirroring CI's own Makefile. |
| FR-SC-7.8 | The generator MUST NOT generate language-specific recipes (no `cargo build`, no `pytest`, no `npm test`). The consumer fills the vacuous targets themselves; the generator's purpose is structural compliance, not stack-specific defaults. |

### 2.5 Generated `config/` Files

#### FR-SC-8: Copy List and Substitutions

| ID | Requirement |
|----|-------------|
| FR-SC-8.1 | The generator MUST copy the following six per-project-overridable config files from `CI/config/` into `<consumer>/config/`: `coverage_thresholds.yaml`, `file_length_limits.yaml`, `dead_code.yaml`, `dependency_excludes.yaml`, `duplicate_dependency_excludes.yaml`, `markdown_docs.yaml`. |
| FR-SC-8.2 | The generator MUST NOT copy any of: `banned_words.yaml`, `banned_words_exceptions.yaml`, any `*_schema.yaml`, `blocked_commit_patterns.yaml`, `required_hooks.yaml`, `silent_swallow_patterns.yaml`, `sensitive_files.yaml`, `boot_layout.yaml`. These are workspace-policy shared across all consumers; copying them creates silent shadow-copies that drift. |
| FR-SC-8.3 | For `coverage_thresholds.yaml`, the generator MUST post-process the copied file by substituting `source_path: ci` for `source_path: <project-name>` (the profile's `project` field). Without this, the coverage checker would scan CI's own source tree from the consumer's project, producing nonsense metrics. |
| FR-SC-8.4 | For `dead_code.yaml`, the generator MUST post-process the copied file by substituting `scan_paths: [ci]` for `scan_paths: [<language-default-dir>]` based on the profile's `languages`. For `rust`, default is `[src]`. For `python`, default is `[]` (no scan paths; the consumer fills them). For `node`, default is `[]`. For `any` (only or in combination with others), default is `[]`. |
| FR-SC-8.5 | When a destination file does not exist, the generator MUST copy without prompting. |
| FR-SC-8.6 | When a destination file exists AND its contents match the canonical CI default (modulo the FR-SC-8.3 / FR-SC-8.4 substitution), the generator MUST copy (effectively a no-op refresh). |
| FR-SC-8.7 | When a destination file exists AND its contents differ from the canonical CI default AND `--force` is NOT set, the generator MUST skip with a `ci_warn` listing the file. The generator MUST NOT silently overwrite a customised config. |
| FR-SC-8.8 | When a destination file exists AND its contents differ from the canonical default AND `--force` IS set, the generator MUST overwrite and MUST print a summary of overwritten files at the end so the consumer can `git diff` and re-apply local edits. |
| FR-SC-8.9 | After every copy, the generator MUST verify the destination file is byte-equal to its source (modulo FR-SC-8.3 / FR-SC-8.4 substitutions). If the copy is corrupted (e.g., disk full), the generator MUST fail fast, list the affected path, and exit non-zero. |

### 2.6 Generated `quality_exceptions.yaml`

#### FR-SC-9: Rendering and Overwrite Policy

| ID | Requirement |
|----|-------------|
| FR-SC-9.1 | The generator MUST render `quality_exceptions.yaml` from `CI/templates/quality_exceptions.template.yaml` by substituting `__PROJECT_NAME__` with the profile's `project` field. |
| FR-SC-9.2 | If `<consumer>/quality_exceptions.yaml` already exists, the generator MUST NEVER overwrite it. The file is reviewer-visible per `templates/quality_exceptions.template.yaml` lines 5-7; overwriting loses exception history. The generator MUST print a `ci_warn` and continue. |
| FR-SC-9.3 | This STRICTER overwrite policy (compared to FR-SC-8.7 / FR-SC-8.8) is intentional: config files are operational tuning; `quality_exceptions.yaml` is social auditing. `--force` does NOT bypass FR-SC-9.2. |

### 2.7 Template Maintenance Mode

#### FR-SC-10: `--emit-template`

| ID | Requirement |
|----|-------------|
| FR-SC-10.1 | `scaffold-ci --emit-template` MUST regenerate `CI/templates/ci-profile.template.yaml` from `CI/config/required_hooks.yaml` and exit. |
| FR-SC-10.2 | The template MUST group hooks by stage (`pre-commit`, `commit-msg`, `pre-push`). Within each stage, hooks MUST be sorted: `safety: true` first, then `mandatory: true` and `safety: false`, then `mandatory: false` (exemptable). Within each tier, the registry's declaring order MUST be preserved (stable sort). |
| FR-SC-10.3 | Each hook in the template MUST be a bare string (`- <id>`), no inline metadata. The template is a starting point, not a runtime config. |
| FR-SC-10.4 | The template MUST include a header comment block explaining: source, regeneration command, and the "DO NOT edit by hand" warning. The template MUST include a footer comment block with totals N (pre-commit=A, commit-msg=B, pre-push=C), timestamp, and source. |
| FR-SC-10.5 | `--emit-template` MUST ignore all other flags (`--consumer`, `--profile`, `--force`, `--dry-run`). It MUST NOT touch disk beyond the single template file. |

### 2.8 Makefile Target and Manifest

#### FR-SC-11: `scaffold-ci` Makefile Target

| ID | Requirement |
|----|-------------|
| FR-SC-11.1 | CI's own `Makefile` MUST define a `scaffold-ci` target accepting `CONSUMER=path` and optional `ARGS="--force"`, `--dry-run`, or `--emit-template`. |
| FR-SC-11.2 | When `CONSUMER` is empty AND `ARGS` is not `--emit-template`, the target MUST fail with a non-zero exit code and a usage diagnostic. |
| FR-SC-11.3 | The target recipe MUST invoke `bash scripts/scaffold-ci --consumer $(CONSUMER) $(ARGS)`. |
| FR-SC-11.4 | The target MUST appear in `make help` output (via the `## Show this help` annotation convention). |

#### FR-SC-12: Manifest Entry

| ID | Requirement |
|----|-------------|
| FR-SC-12.1 | `scripts/manifest.yaml` MUST contain an entry for `scaffold-ci` with the fields specified in SPEC §13. |
| FR-SC-12.2 | The manifest entry MUST include `category: bootstrap`. |
| FR-SC-12.3 | The manifest entry MUST include `make_target: scaffold-ci`. |

### 2.9 Idempotency

#### FR-SC-13: Repeat-Run Behaviour

| ID | Requirement |
|----|-------------|
| FR-SC-13.1 | Running the generator twice with `--force` MUST produce byte-identical output for `.pre-commit-config.yaml` and `Makefile`, modulo the timestamp embedded in the auto-generated file header comments. |
| FR-SC-13.2 | The timestamp comment is the ONLY permissible bytewise drift between two consecutive `--force` runs. This is intentional: it documents when the file was last regenerated. |
| FR-SC-13.3 | Running the generator without `--force` when all destination files match the canonical defaults MUST produce zero filesystem changes (FR-SC-8.6 no-op refresh + FR-SC-9.2 quality_exceptions.yaml never overwritten). |

## 3. Non-Functional Requirements

### 3.1 Platform and Tooling

| ID | Requirement |
|----|-------------|
| NFR-SC-1 | The generator MUST be pure bash + awk. No Python interpreter, no Node runtime, no Rust binary. Sourced libraries restricted to `CI/lib/ci.sh`. The awk parser is a new file `CI/lib/parse_hook_yaml.awk`. |
| NFR-SC-2 | The generator MUST require only `realpath(1)` (coreutils >= 8.23, already a workspace dependency), `awk`, `bash`, `mkdir`, `mktemp` -- all already in the workspace's preflight contract. |

### 3.2 Strict Mode and Error Handling

| ID | Requirement |
|----|-------------|
| NFR-SC-3 | The generator MUST start with `#!/usr/bin/env bash` and `set -euo pipefail`. |
| NFR-SC-4 | The generator MUST NOT pipe to `head` or `tail` (per AGENTS.md Rule 14 -- SIGPIPE masks exit codes). Where a pipeline is used, `PIPESTATUS[0]` MUST be captured on the very next command. |
| NFR-SC-5 | The generator MUST use `source` (not bare `.`) for sourcing `lib/ci.sh` and MUST follow with `|| exit N` (per AGENTS.md Rule 14 -- `sh-bare-source` guard). |
| NFR-SC-6 | The generator MUST NOT swallow errors. Every non-zero exit from a forked subprocess MUST propagate to a generator-level failure diagnostic. No `2>/dev/null`, no `|| true`, no `|| :` except where the failure semantics are documented as "skip this iteration" (e.g., the auto-insertion's `|| continue` on missing optional fields). |

### 3.3 Banned Patterns

| ID | Requirement |
|----|-------------|
| NFR-SC-7 | The generator's source MUST NOT contain any banned words per `CI/config/banned_words.yaml`. The CI banned-words checker will catch violations at commit time. |
| NFR-SC-8 | The generator MUST NOT contain `unsafe` code blocks (not applicable to bash, listed for completeness -- the rule applies across languages). |

### 3.4 File Length and Format

| ID | Requirement |
|----|-------------|
| NFR-SC-9 | All implementation files (the bash script, awk parser, test files) MUST be under 512 lines per `CI/config/file_length_limits.yaml`. Markdown documentation (this document, the SPEC) is exempt by config scope. |
| NFR-SC-10 | The bash script and awk parser MUST use 4-space indentation (matching the existing `scripts/generate-hooks` and `lib/parse_precommit_config.awk` conventions). |

### 3.5 Determinism and Reproducibility

| ID | Requirement |
|----|-------------|
| NFR-SC-11 | The generator MUST be deterministic: same profile + same `required_hooks.yaml` + same configs = same output (modulo the timestamp comment in the auto-generated headers per FR-SC-13.2). |
| NFR-SC-12 | The generator MUST NOT depend on environment variables other than `PATH` (for finding `realpath`) and the `BASH_SOURCE` of the script itself (for resolving `CI_ROOT`). No `HOME`, no `USER`, no `WORKSPACE_ROOT`. |

### 3.6 Security

| ID | Requirement |
|----|-------------|
| NFR-SC-13 | The generator MUST NOT make network calls. No `curl`, no `wget`, no DNS resolution. |
| NFR-SC-14 | The generator MUST NOT execute any code from the profile YAML (the profile is data, not code). The awk parser is a pure tokeniser; no `eval` on YAML contents. |
| NFR-SC-15 | The generated `.pre-commit-config.yaml::entry:` strings are constructed ONLY from values present in `required_hooks.yaml` or the user-supplied `overrides` block (which is audited via PR review of `ci-profile.yaml`). The generator MUST NOT inject any external content into `entry:` values. |
| NFR-SC-16 | If the consumer directory contains a symlink pointing outside the consumer's own subtree, the generator MUST refuse to follow it for the purpose of writing files (writing via `mv` from `mktemp` is safe by construction, but the generator MUST NOT resolve symlinks in the destination path before writing). |

## 4. Test Requirements

### 4.1 Coverage

| ID | Requirement |
|----|-------------|
| TR-SC-1 | The unit test file `tests/unit/test_scaffold_ci.sh` MUST cover every validation phase (A through E), the path-computation function, and every hook-kind entry-render rule (six kinds). |
| TR-SC-2 | The integration test file `tests/integration/test_scaffold_ci.sh` MUST cover: full scaffold run, dry-run, force, emit-template, vendored tier, missing profile, unknown hook ID. Each integration test asserts filesystem state after the run. |

## 5. Acceptance Criteria

The feature is accepted when ALL of the following hold:

1. Files in SPEC §2 ("New Files") exist with the documented contents.
2. Files in SPEC §2 ("Files modified") contain the documented additions.
3. `make -C projects/CI lint && make -C projects/CI type-check &&
   make -C projects/CI test` all pass (full CI self-test suite).
4. `make -C projects/CI scaffold-ci ARGS=--emit-template` regenerates
   `CI/templates/ci-profile.template.yaml` and the regenerated file
   matches the committed version modulo the timestamp footer.
5. For a temp fake consumer directory under `/tmp`, `make -C projects/CI
   scaffold-ci CONSUMER=<temp>` writes all five in-scope files
   (`.pre-commit-config.yaml`, `Makefile`, `config/coverage_thresholds.yaml`,
   `config/file_length_limits.yaml`, `config/dead_code.yaml`,
   `config/dependency_excludes.yaml`,
   `config/duplicate_dependency_excludes.yaml`,
   `config/markdown_docs.yaml`, `quality_exceptions.yaml`) with the
   expected `<REL_CI>` substitution and the expected project-name
   substitution in `coverage_thresholds.yaml` and `dead_code.yaml`.
6. In the temp consumer, `make -C <temp> install-hooks` succeeds and
   writes `.git/hooks/pre-commit`, `.git/hooks/commit-msg`,
   `.git/hooks/pre-push` with executable bits set.
7. In the temp consumer, `make -C <temp> contract-check` exits 0.
8. The unit and integration test files in SPEC §14 exist and pass under
   the existing test harness without modification to the harness.
9. No banned words in any new file (verified by `make check` which
   runs the banned-words checker).
10. All new files are under 512 lines (verified by `ci_check_file_length`
    which runs as part of `make check`).
