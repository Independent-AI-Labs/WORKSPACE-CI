# workspace-ci Hook Generation Guide

## Overview

workspace-ci uses `.pre-commit-config.yaml` as a hook configuration format but does NOT use the `pre-commit` Python framework. Instead, `scripts/generate-hooks` reads the YAML and generates native bash scripts in `.git/hooks/`.

This eliminates the pre-commit framework's stashing behavior (which temporarily removes files from disk, breaking live dev servers) while keeping the familiar configuration format.

## Quick Start

```bash
# From any project that has a .pre-commit-config.yaml:
../CI/scripts/generate-hooks

# Or via Makefile:
make install-hooks
```

This reads `.pre-commit-config.yaml` and writes:
- `.git/hooks/pre-commit`: runs on every commit
- `.git/hooks/pre-push`: runs before push
- `.git/hooks/commit-msg`: validates commit message format

## How It Works

### 1. Parsing

`generate-hooks` invokes `lib/parse_precommit_config.awk` to extract hook records from YAML. Each hook becomes a tab-separated record with fields:

```
id | name | entry | stage | pass_filenames | always_run | files | exclude | args
```

### 2. Grouping

Hooks are grouped by stage: `pre-commit` (default), `commit-msg`, `pre-push`.

### 3. Code Generation

For each stage, a bash script is generated with:

```bash
#!/usr/bin/env bash
# AUTO-GENERATED by ../CI/scripts/generate-hooks -- do not edit.
set -euo pipefail

_ROOT="$(git rev-parse --show-toplevel)"
cd "$_ROOT"
source "${_ROOT}/../CI/lib/ci.sh"
```

Then for each hook in that stage:

1. **File gating**: if `files` regex is set and `always_run` is not `true`, only run when staged files match the regex
2. **Execution**: run the `entry` command, appending `args` if set
3. **Filename passing**: if `pass_filenames` is `true`, matching staged filenames are appended as arguments
4. **Formatter detection**: hooks named `ruff-format` or `ruff` are treated as formatters -- after execution, if files were modified, they're auto-staged and the hook fails (user re-runs commit)

### 4. Unstaged Check

The `ci_check_unstaged` hook (if configured) runs BEFORE any other hook. If unstaged changes exist:
1. Print the diff
2. Auto-stage with `git add -A`
3. Exit 1 with message "Re-run: git commit"

This prevents the pre-commit framework's stashing behavior entirely. Nothing is ever temporarily removed from disk.

## Configuration Format

The `.pre-commit-config.yaml` format is a subset of the pre-commit spec. Only `repo: local` hooks with `language: system` are supported.

### Supported Fields

| Field | Default | Description |
|-------|---------|-------------|
| `id` | (required) | Hook identifier |
| `name` | `""` | Display name (printed during execution) |
| `entry` | (required) | Shell command to execute |
| `language` | `system` | Must be `system` (only supported value) |
| `stages` | `[pre-commit]` | When to run: `pre-commit`, `commit-msg`, `pre-push` |
| `pass_filenames` | `true` | Append matching staged filenames as arguments |
| `always_run` | `false` | Run even if no staged files match `files` regex |
| `files` | `""` | Regex filter -- only run when staged files match |
| `exclude` | `""` | Regex exclude -- skip files matching this pattern |
| `args` | `[]` | Extra arguments appended to `entry` |

### Unsupported Fields (ignored)

`types`, `types_or`, `verbose`, `require_serial`, `additional_dependencies`, `minimum_pre_commit_version`, `repo` (anything other than `local`).

## Complete Example

```yaml
repos:
  - repo: local
    hooks:
      # --- Gate: unstaged changes ---
      - id: check-unstaged
        name: Block Unstaged Changes
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_check_unstaged'"
        language: system
        pass_filenames: false
        always_run: true

      # --- Formatters (modify files, trigger re-commit) ---
      - id: ruff-format
        name: ruff format
        entry: uv run ruff format --config ../../res/config/ruff.toml .
        language: system
        pass_filenames: false
        always_run: true
      - id: ruff
        name: ruff lint
        entry: uv run ruff check --config ../../res/config/ruff.toml --fix .
        language: system
        pass_filenames: false
        always_run: true

      # --- Type checker ---
      - id: mypy
        name: mypy
        entry: uv run mypy --config-file ../../res/config/mypy.toml src
        language: system
        pass_filenames: false
        always_run: true

      # --- shared checks ---
      - id: block-sensitive-files
        name: Block Sensitive Files
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_block_sensitive_files'"
        language: system
        pass_filenames: false
        always_run: true
      - id: check-dependency-versions
        name: Check Dependency Versions
        entry: uv run python -m ci.check_dependency_versions
        language: system
        pass_filenames: false
        always_run: true
      - id: check-banned-words
        name: Check Banned Words
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_check_banned_words'"
        language: system
        pass_filenames: false
        always_run: true
      - id: check-file-length
        name: Check File Length (max 512 lines)
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_check_file_length'"
        language: system
        pass_filenames: false
        always_run: true
      - id: check-init-files
        name: Enforce empty __init__.py
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_check_init_files'"
        language: system
        pass_filenames: false
        always_run: true

      # --- Commit message checks ---
      - id: check-commit-message
        name: Check Commit Message Format
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_check_commit_message \"$1\"' --"
        language: system
        stages: [commit-msg]
      - id: block-coauthored
        name: Block Co-authored Commits
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_block_coauthored \"$1\"' --"
        language: system
        stages: [commit-msg]

  # --- Pre-push checks ---
  - repo: local
    hooks:
      - id: block-coauthored-history
        name: Block Commits with Blocked Patterns (full history scan)
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_block_coauthored_history'"
        language: system
        stages: [pre-push]
        pass_filenames: false
      - id: verify-coverage
        name: Verify Test Coverage
        entry: "bash -c 'source ../CI/lib/checks.sh && ci_verify_coverage'"
        language: system
        stages: [pre-push]
        pass_filenames: false
```

## Hook Stages

| Stage | When | Hooks |
|-------|------|-------|
| `pre-commit` | Before each commit | unstaged check, formatters, linters, type checker, banned words, sensitive files, file length, init files, dependency versions |
| `commit-msg` | After writing commit message | commit message format, co-authored-by blocking |
| `pre-push` | Before pushing to remote | full history scan for blocked patterns, test coverage |

## Commit scope rule

**One concern per commit.** A single commit should answer a single "why"
question. Examples of legitimate bundling and illegitimate bundling:

- ✅ Renaming a symbol and updating every caller: one concern
  (the rename).
- ✅ Adding a feature and the test that proves it works: one concern
  (the feature is not done without the test).
- ✅ Deleting a module and the doc references that pointed at it: one
  concern (the cleanup).
- ❌ Adding a feature *and* bumping a dependency *and* reformatting an
  unrelated file: three concerns, three commits.
- ❌ "Big refactor" commits that mix a module split with behaviour
  changes: the split and the behaviour change should be separate
  commits so `git blame` and `git bisect` stay useful.

When in doubt: would a reviewer want to `git revert` the behaviour
change but keep the refactor? If yes, they belong in separate commits.

## Formatter Behavior

Hooks with `id` matching `ruff-format` or `ruff` are classified as formatters. After a formatter runs:

1. Check if any files were modified (`git diff --quiet`)
2. If modified: auto-stage changes, print message, exit 1
3. User re-runs `git commit` -- second run has nothing to format, passes cleanly

This two-pass pattern avoids the need for stashing.

## Makefile Integration

```makefile
install-hooks: preflight
	@bash ../CI/scripts/cleanup-precommit 2>/dev/null || true
	bash ../CI/scripts/generate-hooks

check-hooks: preflight
	bash ../CI/scripts/generate-hooks --dry-run
```

## Cleanup

To remove the pre-commit Python framework from a project:

```bash
../CI/scripts/cleanup-precommit
```

This removes:
- `~/.cache/pre-commit/` (framework cache)
- `pre-commit` Python package (pip/uv)
- Old `.git/hooks/pre-commit` generated by pre-commit install

It does NOT remove `.pre-commit-config.yaml` -- that file is now the configuration source for `generate-hooks`.

## Migration from pre-commit

1. Keep your `.pre-commit-config.yaml` as-is
2. Run `../CI/scripts/cleanup-precommit` to remove the framework
3. Run `../CI/scripts/generate-hooks` to install native hooks
4. Update your `Makefile` `install-hooks` target (see above)
5. Remove `pre-commit` from `pyproject.toml` dependencies

## Recovering from "hook auto-staged a file you didn't want"

Some checks (notably `Block Unstaged Changes`) auto-stage a file they
detect as unstaged. If you started a commit and the hook pulled in a
file you did not intend to commit, you may find `git reset` and
`git rm --cached` both blocked by the local git-guard.

Workaround:

```bash
# Unstage a brand-new file without removing it from the working tree
# (git-guard does not block update-index).
git update-index --force-remove <path>

# For a modified-but-tracked file, restore the index entry from HEAD
git update-index --cacheinfo 100644,$(git rev-parse HEAD:<path>),<path>
```

If the auto-staged file is a build/test artefact (`.coverage`,
`__pycache__/`, etc.), add it to `.gitignore` in the same commit or a
companion commit so the hook stops re-capturing it.

Never bypass the hook with `git commit --no-verify`: it is blocked by
the git-guard, and even if it were not, the hook is almost certainly
catching a real problem.
